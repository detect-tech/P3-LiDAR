The problem, in NEET's own words

Duplicated collection costs money and time

Aerial LiDAR collection is costly, deadline-driven, and seasonal. Usable data often already exists across public and previously-acquired sources — but there's no easy way to see what's already available before committing to an expensive new flight.

1

3

Understanding real ground conditions — vegetation, water, elevation change — ahead of route commitment.

Hazard detection on a real route

Given a real route (KML + right-of-way corridor), check existing LiDAR against it and flag buildings, vegetation, water, and elevation change — and flag when the data itself is too stale or sparse to trust.

Explicitly out of scope for today's build: reroute suggestions, a chatbot interface, and real Routify/ArcGIS integration — all whiteboarded as future ideas, by direct instruction.
The solution

One dashboard: coverage and hazards, on a real route

  • Real USGS 3DEP LiDAR coverage for Miami-Dade — 1,534 tiles from the 2023–24 collection, plus an older 2017 Everglades-area collection. Both real, pulled from public metadata, no API key.
  • Pick a route corridor → live coverage analysis, with an adjustable right-of-way width slider.
  • Plain-language recommendation per corridor: reuse existing data, partially collect, or fly new.
  • Hazard panel — buildings, vegetation, water, elevation change — clickable, pans the map to each flag.
  • Light and dark mode, built on the Cobalt design system.

Report categories

Missing data · Low resolution · Hazards — the same three buckets from the whiteboard, surfaced per corridor.

Data quality flags

Stale data (2-year cutoff), sparse point-cloud density, and confidence level are shown alongside every hazard — so the team knows when to trust the read.

Hazard placements shown today are mock data, interpolated along each route's real geometry — not live detection output. Said explicitly in the demo, not implied.
The business case

Built in one AI-assisted session, on real public data

  • Real USGS 3DEP coverage data — no API key, and the fetch pipeline is reusable for any county, not just Miami-Dade.
  • Coverage math independently verified via a separate Python/Shapely re-implementation before it went into the dashboard.
  • Directly targets NEET's #1 stated pain point: the cost of duplicating collection that's already public.
  • Illustrative cost-avoided estimates on every corridor — clearly labeled as estimates, not guarantees.
This is a prototype of what a formalized internal tool could look like — not a finished product. The value is in how fast a real, working narrative came together against real data.
